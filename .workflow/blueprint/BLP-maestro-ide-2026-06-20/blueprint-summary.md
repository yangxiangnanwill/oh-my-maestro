## Architecture

```
Browser (SvelteKit) ←→ WebSocket ←→ Node.js Server ←→ CLI Subprocess (maestro / Claude Code)
                                              ↑
                                        EventBus + State Sync Engine
```

- **Backend**: Node.js + Hono + WebSocket + node-pty + chokidar
- **Frontend**: SvelteKit + xterm.js + WebSocket client
- **Integration**: CLI subprocess → adapter layer → structured events → WebSocket → UI

## Delivery Phases

1. **Foundation** — State Sync Engine + Concept Translator (unblocks all)
2. **Core MVP** — Project Radar + Workflow Commander (delivers core value)
3. **Interaction** — Terminal Bridge + AI Dialog (enhances accessibility)
4. **Trust** — Approval Gate (completes safety model)

## Critical Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Platform | Local web app | Fastest MVP, cross-platform, full UI freedom |
| AI Integration | CLI subprocess first | Zero API dependency, fastest validation |
| Frontend | SvelteKit | Compile-time reactivity, WebSocket-friendly, small bundle |
| State Sync | Event-driven + WebSocket | Only option meeting 500ms latency + bidirectional needs |

## Risks

1. **CLI output parsing fragility** — Mitigated by versioned adapter registry
2. **Concept abstraction leak** — Mitigated by unified defense (SA middleware + UI translator + UX rendering rules)
3. **Windows ConPTY stability** — Mitigated by platform-specific PTY control strategies
4. **MVP scope creep** — Mitigated by strict feature boundary (7 features, no additions)
